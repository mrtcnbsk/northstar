# Wave 6 — Memory & Learning Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Branch: `feat/wave-6-memory` (off main; Waves 0-5 merged). `bun` at `~/.bun/bin`; never stage `bun.lock`; sweeps via `bun run script/test-runner.ts`; clean `"$(getconf DARWIN_USER_TEMP_DIR)"opencode-test-*` after runs. Everything must be TESTABLE WITHOUT an embedder key (mock `IEmbedder`/`IVectorStore` for RAG; lexical/tmpdir for memory) — the org-RAG's real vector search needs a BYOK embedder at runtime but the code + tests use stubs and DEGRADE GRACEFULLY when no key is configured.

**Goal:** Make the org LEARN across runs: every completed/halted run writes a deterministic postmortem to `.kilo/org/lessons.md` + an org-scoped memory pool; later runs warm-start on those lessons; and prior runs' deliverables become searchable (org-scoped RAG with citations) so an agent can answer "how did we solve X last run?"

**Architecture:** REUSE the existing substrate — do not rebuild. `packages/kilo-memory`'s `Memory.*` facade is `{root}`-parameterized, so an ORG memory pool is the same engine pointed at `.kilo/org/memory` (lexical recall, keyless). `packages/kilo-indexing`'s RAG (embedder iface + LanceDB + `directoryPrefix` namespace + open-ended payload) is reused via constructor injection (stub embedder in tests). The postmortem is a DETERMINISTIC summary of `OrgState.Run` (no LLM in v1) written at the run's completion choke point (`OrgAdvanceTool` `batch.done`/halt branches, inside `withRunLock`), BEST-EFFORT (a postmortem failure must never break run completion).

**Tech Stack:** `Memory.*` (`packages/kilo-memory/src/memory.ts`), `KiloIndexing`/`CodeIndexSearchService`/`LanceDBVectorStore` (`packages/kilo-indexing`), `OrgState`/`OrgAudit`/`OrgArtifacts`, `OrgAdvanceTool`/`KiloToolRegistry`, `semantic_search` tool.

**Grounding (verified, Wave-6 exploration 2026-07-11):**
- Session memory: `kilo_memory_save`/`kilo_memory_recall` → `Memory.*` (`memory.ts`), `{root}`-parameterized; stores markdown `- key :: value` + JSON state; recall is PURE LEXICAL (keyword scoring, `recall/recall.ts`), NO embeddings; `scope` hardcoded `"project"` (`schema.ts` L52/L191) — no org/cross-run/dept scope today. Tests: tmpdir root, no key (`kilo-memory/test/memory.test.ts`).
- RAG: `semantic_search` (`tool/semantic-search.ts`, params `query`+`path` where `path`=directoryPrefix namespace; output formatter L94-100 already prints `filePath:startLine-endLine`) → `KiloIndexing.search(query, directoryPrefix?)`. Embedder BYOK (`kilo-indexing/src/indexing/interfaces/embedder.ts`, 10 providers); LanceDB local (`lancedb-vector-store.ts`, single collection keyed by workspace path; `directoryPrefix` = filePath LIKE-prefix; `Payload` open-ended → can carry `runID`/`dept`). Constructor-injected → testable with mock embedder+store (`test/kilocode/indexing/search-service.test.ts`).
- Run completion choke point: `OrgAdvanceTool.execute` `batch.done` branch (`tools.ts` ~L196-198), inside `withRunLock` (L177), has `dir`/`org`/`run_id`. Halt choke points: `batch.halted`, `OrgDecisionTool` no-go, `OrgStopTool`. `advance` flips `status="completed"` once (`runner.ts` L702-705; re-entry early-exits at L504). Run data: `OrgState.Run` (idea, mode, stages{status,costs,attempts,incompleteAttempts,decision,decisionNote,startedAt,completedAt}, haltReason), `OrgState.runSummary`, `OrgState.stageCost`, `OrgAudit.read`. `.kilo/org/lessons.md` + `.kilo/org/memory/` are GREENFIELD.
- Org tool registration: `KiloToolRegistry.{infos,build,extra}` (`registry.ts` L86-90/192-196/313-317) + `applyVisibility` `org_*` gate (L395, needs `.kilo/organization.jsonc`) + primary-mode gate (L250) + per-tool `guardCeo` (`tools.ts` L77) + `withRunLock`.
- Lifecycle template (if decoupling): `MemoryLifecycle.subscribe` (`kilocode/memory/turn.ts`) on `TurnClose`, wired in `bootstrap.ts` L45. No org-run-complete Bus event exists yet.

**Locked design decisions:**
1. **Deterministic postmortem (no LLM in v1).** The postmortem is a structured summary of run state (per-stage status/cost/attempts/decision, total cost, halt reason, gate decisions) — reliable + testable. LLM-narrated qualitative lessons = follow-up (needs a model + is non-deterministic).
2. **Best-effort hook.** The postmortem runs inside `withRunLock` at the completion/halt choke points but is wrapped so ANY failure logs + continues — it must NEVER break run completion. Fires once per end-state (completion re-entry early-exits; halt is terminal).
3. **Org memory = the existing engine at a new root** (`.kilo/org/memory`, project-local/committable, cross-run). Lexical recall (keyless). A lightweight `dept` tag rides in the record. New `org_memory_save`/`org_recall` tools (CEO-scoped, `org_*` visibility).
4. **Org-RAG namespace = payload `runID`/`dept` + `directoryPrefix` over deliverables**, reusing the injected engine; stub embedder in tests; **degrades gracefully** (no embedder key → search returns an empty/"unavailable" result, never throws). Citations added to the shared `semantic_search` formatter.

**Protocol:** per task — fresh implementer, spec review, quality review; Important findings fixed + re-reviewed; TDD. Separate commits. `tracked-followups.md` continues. Wave-close adversarial review before merge (focus: the postmortem best-effort invariant + no double-write, the org-memory root isolation, the RAG graceful-degradation-without-key).

### W6.2 — Postrun postmortem hook → lessons.md + org memory

**Files:** new `src/kilocode/organization/postmortem.ts` (a PURE builder `OrgPostmortem.build(run, summary, audit): string` + a `write(projectDir, run, ...)` that appends to `.kilo/org/lessons.md`), modify `tools.ts` (`OrgAdvanceTool` `batch.done` + `batch.halted` branches; `OrgDecisionTool` no-go; `OrgStopTool`) to call it best-effort inside the run lock, plus save a compact lesson to the W6.1 org pool; tests.
`OrgPostmortem.build(run, OrgState.runSummary(run), audit)` → a deterministic markdown section: run id + idea + final status (+ haltReason), a per-stage table (stage | status | cost | attempts | decision), total cost, gate decisions from the audit, and a one-line outcome (shipped / halted-at-X / no-go). `write(...)` APPENDS this section to `.kilo/org/lessons.md` (create if absent; use `Filesystem.write`/append like `state.ts`; each run appends one section, newest last or a bounded ring — decide + document). In `tools.ts`, at each end-state choke point, call `OrgPostmortem.write(...)` AND `OrgMemory` save (a compact "run <idea>: <outcome>, $<cost>, <key lesson>" entry, dept-tagged by the failing/last stage) — ALL wrapped in try/catch that logs and continues (a postmortem failure MUST NOT change the tool's returned action or fail completion). Ensure it fires ONCE per run end (completion re-entry at `runner.ts` L504 early-exits before the done branch re-runs — verify; guard against double-append if the same end-state advance is called twice, e.g. a `postmortemWritten` flag on run state OR an idempotent check that lessons.md already has this runID's section). Tests: a completed multi-stage run → `lessons.md` gains a section with the right per-stage data + total cost; a halted (no-go / budget) run → captures the failure + haltReason; the postmortem builder is pure (unit-test the markdown from a fabricated run); a THROWING postmortem (e.g. unwritable path) does NOT break `org_advance`'s done/halted result (inject a failing writer, assert the action still returns); no double-append on re-entry.
