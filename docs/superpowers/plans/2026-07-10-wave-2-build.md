# Wave 2 — Build & Test Runtime Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Branch: `feat/wave-2-build` (off main; Waves 0+1 merged). `bun` at `~/.bun/bin`; never stage `bun.lock`; sweeps via `bun run script/test-runner.ts`; clean `"$(getconf DARWIN_USER_TEMP_DIR)"opencode-test-*` after runs. Tools must be TESTABLE WITHOUT Xcode (mock the spawner / use fixture output) but usable on a Mac with Xcode.

**Goal:** Let the org autonomously build, lint, and test a real SwiftUI app and get STRUCTURED results back — turning xcodebuild's thousands of noisy lines into `{ok, errors[], warnings[]}` so the build/test loop works within the Wave-1 budget instead of drowning agents in output.

**Grounding (verified read-only):** shell tool (`src/tool/shell.ts`) already spawns xcodebuild/swift/simctl via `ChildProcessSpawner`; dev/test/debug workers already allow `xcodebuild*`/`swift build*`/`swift test*`/`xcrun simctl*`; `BashArity` (`src/permission/arity.ts`) has `swift: 2` and is extensible; `background-process` tool handles long-running processes (simulators) with ready-detection; sandbox is OFF for org workers so exec already works; SourceKit-LSP is NOT wired (out of scope — shell error-parsing suffices). New kilocode tools register in `KiloToolRegistry.infos()/build()/extra()` (`src/kilocode/tool/registry.ts`).

**SNR scope decisions:** Build/test/lint/simulator run via the EXISTING shell + allowlists — Wave 2 adds STRUCTURED-OUTPUT tools (the leverage), not shell re-wrappers. **CUT for v1:** log aggregation (shell + background-process suffices; deferred), SourceKit-LSP (shell parsing covers Swift diagnostics). Dedicated new tools: `xcode_build`, `xcode_test`, `crash_symbolicate`. Lint stays config-first (arity + agent Do-lines) unless output parsing proves needed.

**Protocol:** per task — fresh implementer, spec review, quality review; Important findings fixed + re-reviewed; TDD. Separate commits.

### W2.2 — `xcode_build` structured build tool

**Files:** Create `src/kilocode/tool/xcode-build.ts`; wire in `registry.ts`; add a permission key; `test/kilocode/tool/xcode-build.test.ts`.
Tool `xcode_build`: params `{ scheme?, workspace?, project?, configuration?, destination?, extraArgs? }`. Runs `xcodebuild build ...` via the SAME spawner/exec path the shell tool uses (reuse `ChildProcessSpawner`/the shell machinery — do NOT reimplement process spawning; study shell.ts and extract or call the shared primitive). Parses xcodebuild output into `{ ok: boolean, errors: [{file?, line?, column?, message}], warnings: [...], durationMs, rawLogPath }` (write the full raw log to a temp/state file, return only the structured summary + path so agents don't ingest 10k lines). Parser: match xcodebuild's `file:line:col: error:` / `warning:` lines + the final `** BUILD SUCCEEDED/FAILED **`. Permission: gate on a new `xcode_build` permission key (or reuse `shell` semantics — decide and justify; likely its own key so workers can be granted it precisely). Available to primary + subagents (workers need it).
**Testing without Xcode:** inject the exec primitive so tests feed FIXTURE xcodebuild output (a real captured xcodebuild-failure log + a success log as fixtures) and assert the parse. Do NOT require Xcode in the test.
Tests: parses a failure log into specific {file,line,message} errors; parses success → ok:true; malformed/empty output → ok:false with a clear reason; raw log written + path returned. Report the injection seam you used.

### W2.5 — `crash_symbolicate` tool

**Files:** Create `src/kilocode/tool/crash-symbolicate.ts`; wire registry; test.
Tool `crash_symbolicate`: params `{ crashLog: string (or path), dsymPath: string }`. Wraps `atos`/`llvm-symbolizer` (via the shared exec primitive) to map addresses → symbol/file:line; returns the symbolicated stack trace text + `{framesResolved, framesTotal}`. Parse the crash log for the process's frames + load address, run the symbolizer per frame (or batch). Degrade gracefully when dsym missing / symbolizer absent (return the raw trace + a note, don't fail hard). Fixture test: a raw crash-log fixture + a stubbed symbolizer exec returning known symbols → assert the resolved trace; missing-dsym → graceful note. This satisfies the wave exit criterion's "seeded crash → symbolicated trace."
