# Wave 7 — Apple App Store Delivery Pipeline Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Branch: `feat/wave-7-apple` (off main; Waves 0-6 merged). `bun` at `~/.bun/bin`; never stage `bun.lock`; sweeps via `bun run script/test-runner.ts`; clean `"$(getconf DARWIN_USER_TEMP_DIR)"opencode-test-*` after runs. EVERYTHING must be TESTABLE WITHOUT live Apple credentials — inject `fetch` (ASC client + monitor), a fake `ChildProcessSpawner` (archive/export), an in-test EC keypair (JWT), and pure functions (metadata validation). **PUBLIC-REPO SECURITY RULE: the ASC credential (issuer id + key id + `.p8` private key) MUST live ONLY in the out-of-repo auth store (`Global.Path.data/auth.json`, 0o600) or user env vars — NEVER in `.kilo/`, `organization.jsonc`, an agent `.md`, a test fixture, or any committed file. No hardcoded keys, ever.**

**Goal:** Give the org a fixture-tested, BYO-credentials App Store delivery pipeline: archive → IPA export → validate metadata → (with the user's own ASC API key) upload the build to TestFlight, post the marketing metadata, submit for review, and monitor the review state — integrated as a human-gated `delivery` stage so any user who clones this public repo can plug in their own Apple account locally and ship, with no manual steps beyond gate approvals.

**Architecture:** Reuse every substrate. Build artifacts reuse the W2 xcodebuild tool skeleton (`ChildProcessSpawner` + bounded streaming parser); the ASC credential reuses the BYOK pattern (`Auth.Service` + a pure `resolveAscAuth` precedence resolver, exactly like `resolveKiloIndexingAuth`); the REST client mirrors the kilo-gateway fetch+retry+zod pattern with an injectable `fetch`; the ES256 JWT is hand-rolled with `node:crypto` (no new dep); review-monitoring uses the existing `BackgroundJob` + `Bus`; and delivery is a pipeline STAGE reusing the W4 DAG + W5 `gate:"human"`/`haltOn:"no-go"` + the runner's revise loop (rejected metadata → `org_decision "revise"` → re-instruct). No new orchestration primitive; no engine change.

**Tech Stack:** `Tool.define` + `KiloToolRegistry`, `ChildProcessSpawner`, `Auth.Service` (`packages/opencode/src/auth`), `node:crypto` (ES256), `fetch`, `BackgroundJob` (`packages/opencode/src/background/job.ts`), `Bus`, the org runner/schema, org-template jsonc + agents.

### W7.1 — Shared `runXcodebuild` primitive + `xcode_archive` + `ipa_export` tools

**Files:** new `src/kilocode/tool/xcodebuild-exec.ts` (extract the shared spawn→stream→bounded-parse→timeout-race→lazy-sink primitive from `xcode-build.ts`/`xcode-test.ts`, W2-R2), new `src/kilocode/tool/xcode-archive.ts` + `.txt`, new `src/kilocode/tool/ipa-export.ts` + `.txt`; refactor `xcode-build.ts`/`xcode-test.ts` to use the primitive (keep their behavior + tests byte-green); parameterize `StreamingXcodeParser`'s success/fail markers; `config/permission.ts` (+`xcode_archive`,`ipa_export`); `registry.ts` (3 sites); tests.
Extract `runXcodebuild(spawner, trunc, args, { cwd, timeoutMs, successMarker, failMarker })` returning `{ parser, exitCode, rawLogPath, rawLogNote, spawnFailed }`. `xcode_archive` params `{ scheme, workspace?/project?, configuration?, archivePath, extraArgs? }` → `xcodebuild archive -scheme … -archivePath <archivePath>` (marker `** ARCHIVE SUCCEEDED/FAILED **`). `ipa_export` params `{ archivePath, exportOptionsPlist, exportPath, extraArgs? }` → `xcodebuild -exportArchive -archivePath … -exportOptionsPlist … -exportPath …` (marker `** EXPORT SUCCEEDED/FAILED **`); on success, report the produced `.ipa` path(s) found under `exportPath`. **Path-param validation:** the typed path params (archivePath/exportPath/exportOptionsPlist) are validated within-project (relative, no `..`, no absolute) by a small helper — but keep `validateExtraArgs` for `extraArgs` (it would wrongly reject the intrinsic `-archivePath` if these went through extraArgs). Same `{ok, status, errors[], warnings[], rawLogPath}` result shape as xcode_build; `spawn_failed`/`invalid_args` distinct. Grant to the (future) delivery worker; available to subagents. **Tests (fixture, NO Xcode):** fake `ChildProcessSpawner` feeding captured `** ARCHIVE SUCCEEDED **` / failure logs → parsed structured result; a timeout via `TestClock`; a bad path param → `invalid_args`; `xcode-build.test.ts`/`xcode-test.test.ts` stay green after the refactor (the primitive is behavior-preserving).

### W7.2 — BYO-ASC credential resolution + ES256 JWT signer

**Files:** new `src/kilocode/asc/auth.ts` (`resolveAscAuth` + `AscCredential` type), new `src/kilocode/asc/jwt.ts` (`signAscJwt`), tests. NO new npm deps.
`resolveAscAuth({ config, auth, env }): AscCredential | undefined` — a PURE precedence resolver mirroring `resolveKiloIndexingAuth`: read the `Api` record for provider `"appstore-connect"` from the auth store (`key` = the `.p8` PEM, `metadata = { issuerId, keyId }`), falling back to env `ASC_ISSUER_ID` / `ASC_KEY_ID` / (`ASC_PRIVATE_KEY` inline PEM OR `ASC_KEY_PATH` file). Returns `{ issuerId, keyId, privateKeyPem }` or `undefined` when incomplete (→ callers show the "configure your ASC key" message). NEVER logs the private key. `signAscJwt(cred, { now? }): string` — hand-rolled ES256 with `node:crypto`: header `{alg:"ES256", kid: cred.keyId, typ:"JWT"}`, claims `{iss: cred.issuerId, iat, exp: iat+1200, aud:"appstoreconnect-v1"}`, `signingInput = b64url(header)+"."+b64url(claims)`, sig = `crypto.sign("sha256", Buffer.from(signingInput), { key: crypto.createPrivateKey(cred.privateKeyPem), dsaEncoding: "ieee-p1363" })`, token = `signingInput + "." + b64url(sig)`. Inject `now` for determinism. **Tests (NO real key):** generate an EC P-256 keypair in-test (`crypto.generateKeyPairSync("ec", { namedCurve: "P-256" })`, export the private key to PKCS#8 PEM = a synthetic `.p8`); `signAscJwt` produces a token whose header/claims base64url-decode to the expected values, `exp-iat===1200`, `aud="appstoreconnect-v1"`, and whose signature `crypto.verify`s against the public key with `dsaEncoding:"ieee-p1363"`; `resolveAscAuth` precedence (auth store beats env; incomplete → undefined; env-only path works); a test asserting the resolver/ signer never include the PEM in any returned/thrown/logged string except the token itself. **Confirm no secret reaches a committed file** (this module only reads from the auth store / env).

### W7.3 — Direct ASC REST client + core operations

**Files:** new `src/kilocode/asc/client.ts` (`AscClient`), new `src/kilocode/asc/operations.ts` (the delivery ops over the client), tests. Injectable `fetch`.
`AscClient({ credential, fetch = globalThis.fetch, baseUrl = "https://api.appstoreconnect.apple.com", now? })`: builds a fresh JWT per request window (or caches until `exp`), `request(method, path, body?)` sets `Authorization: Bearer <jwt>`, `Content-Type: application/json`, parses JSON, maps non-2xx to a typed `AscError { status, code, title, detail }` (parse ASC's `errors[]` envelope), retries 429/5xx with capped backoff (reuse the `retryable` shape). `operations.ts` — thin, fixture-testable wrappers for the delivery flow: `getAppByBundleId(bundleId)`, `listBuilds(appId)` / `buildProcessingState`, `uploadBuild` (NOTE: binary upload uses the `altool`/`iris` path, not JSON — for v1 scaffold, wrap `xcrun altool --upload-app` via the shared spawner OR document that build upload is the one op needing altool; the ASC JSON API handles metadata/submission/state, not the binary — be explicit about this split), `ensureAppStoreVersion(appId, version)`, `updateVersionLocalizations(versionId, metadata)` (name/subtitle/description/keywords/promo per locale POST/PATCH), `createReviewSubmission`/`submitForReview(versionId)`, `getReviewState(versionId)`. Every op takes the client so tests inject a fake `fetch` returning canned ASC JSON. **Tests (fake fetch, NO Apple):** each op builds the right METHOD+URL+body and the request carries `Authorization: Bearer <token>`; a 429 retries then succeeds; a 4xx maps to a typed `AscError` (not a throw of a raw Response); the metadata PATCH body matches ASC's `{data:{type, id, attributes}}` shape. Report the binary-upload-via-altool split honestly.

### W7.4 — `asc_metadata_validate` tool (Apple length/locale limits)

**Files:** new `src/kilocode/tool/asc-metadata-validate.ts` + `.txt`; `config/permission.ts` (+`asc_metadata_validate`); `registry.ts` (3 sites); grant to `marketing-chief` + `appstore-review-validator` + delivery agents; tests.
Pure `validateAscMetadata(entries: Array<{ locale, name?, subtitle?, promotionalText?, keywords?, description? }>): { ok, violations: [{ locale, field, message }] }` — static ASC limits: name ≤30, subtitle ≤30, promotionalText ≤170, keywords field ≤100 (the comma-joined string), description ≤4000; a required-field check (name + description present per primary locale); a static ASC/BCP-47 locale-code allowlist (unknown locale → violation). `Tool.define("asc_metadata_validate", …)` mirroring `ats_check`: params `{ metadataPath }` (read the deliverable JSON/markdown block) OR `{ entries }` inline; `ctx.ask`; run the pure checker; `{title, output: JSON, metadata:{ok}}`; fail-closed on unparsable input. **Tests (fixtures):** an entry over each limit → the specific violation; an unknown locale → violation; all-within → ok:true; empty/malformed → invalid (fail-closed). Marketing-chief's prompt gains one Do-line: run `asc_metadata_validate` before declaring the listing package READY.

### W7.5 — Review-monitoring background job

**Files:** new `src/kilocode/asc/review-monitor.ts` (the poll-loop `run` Effect + `startReviewMonitor` via `BackgroundJob`), a `BusEvent.define("asc.review.state", …)`, tests.
`startReviewMonitor(deps, { versionId, client, pollMs = 60_000, deadlineMs })` → `BackgroundJob.start({ type:"asc-review-monitor", title, metadata:{versionId}, run })` where `run` loops: `client` → `getReviewState(versionId)` → on a change from the last state, `Bus.publish(AscReviewState, { versionId, from, to })` → `Effect.sleep(pollMs)` → repeat until a TERMINAL state (`PENDING_DEVELOPER_RELEASE` | `READY_FOR_SALE` | `REJECTED` | `DEVELOPER_REJECTED` | `METADATA_REJECTED`) or the overall deadline; return the final state string. Terminal `REJECTED`/`METADATA_REJECTED` carries Apple's rejection reason (fetch the resolution/notes) so the delivery stage can revise. **Tests (fake fetch + `TestClock`):** a scripted `WAITING_FOR_REVIEW → IN_REVIEW → REJECTED` sequence → the job publishes 2 state-change events and resolves with `REJECTED` + reason; `TestClock.adjust` drives the poll sleeps deterministically; a deadline-exceeded path returns a timeout state without hanging; a fetch error is retried/surfaced, never crashes the job.
