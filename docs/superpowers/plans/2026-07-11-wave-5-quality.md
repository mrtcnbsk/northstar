# Wave 5 — Quality Gate Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Branch: `feat/wave-5-quality` (off main; Waves 0-4 merged). `bun` at `~/.bun/bin`; never stage `bun.lock`; sweeps via `bun run script/test-runner.ts`; clean `"$(getconf DARWIN_USER_TEMP_DIR)"opencode-test-*` after runs. Compliance tools must be TESTABLE WITHOUT Xcode/a real app (fixture files); no new npm deps (they'd churn bun.lock) — parse plist/source with self-contained code.

**Goal:** Insert a `review` quality gate before the terminal marketing stage: a review department whose chief fans out parallel reviewer agents (privacy, accessibility, App-Store compliance, security, senior-engineer), each backed by structured compliance tools, aggregating a consensus verdict (per-reviewer votes) that surfaces at a human gate — so a seeded defect (hardcoded secret / missing privacy manifest) blocks the pipeline before ship.

**Architecture:** Reuse everything that exists. The runner's `gate:"human"` + `org_decision(no-go)` halt already implement "block"; W4's parallelism + chief→worker delegation already implement "parallel reviewers." So this wave is: (1) **structured compliance tools** (the leverage — mirror the W2 xcode-tool pattern: parse → `{ok, violations[]}`), (2) **config** — a `review` department + stage + a `review-chief` + 2 missing reviewer agents, wiring the 8 EXISTING validators, (3) a **consensus deliverable convention** (chief prompt: spawn reviewers in parallel, aggregate per-reviewer votes into `review.md`). No new runner/schema code.

**Tech Stack:** `Tool.define` + `KiloToolRegistry` (`packages/opencode/src/kilocode/tool/`), org-template jsonc + agent markdown, the existing `gate:"human"`/`haltOn:"no-go"` runner path.

**Grounding (verified, Wave-5 exploration 2026-07-11):**
- Pipeline (8 stages): evaluation(gate,haltOn)→planning→ux→{backend,frontend}(requires ux)→testing(requires both)→**debugging**→marketing(gate, terminal). **Insert `review` between debugging and marketing.**
- EXISTING validators in `org-template/agents/` (REUSE): `privacy-manifest-validator`, `accessibility-validator`, `appstore-review-validator`, `entitlement-validator`, `hig-validator`, `api-availability-validator`, `swift6-migration-validator`, `localization-validator` — all `mode:subagent`, `model:anthropic/claude-sonnet-5`, workers (no subordinates), `permission:{edit:deny,bash:deny,websearch:allow,webfetch:allow}`. **MISSING:** `review-chief`, `security-validator`, `senior-engineer-reviewer`.
- Structured-tool recipe: `Tool.define("id", Effect.gen(...))` with `{description(from .txt), parameters(Schema.Struct), execute}`; `execute` calls `ctx.ask({permission:"id", patterns, always})`; register in `registry.ts` `infos()` (yield the tool) + `build()` (add to the `Tool.init` map); grant via agent frontmatter `permission:{ id: allow }`. `test/kilocode/organization/template.test.ts` asserts EXACT per-worker tool grants (`BUILD_TOOL_GRANTS`) + agent count (58) + pipeline length (8) + chief `task` perms + "chief prompt mentions a consultant".
- Deliverable validation (`artifacts.ts`): `validate` = file at `deliverables/<stage>.md` exists AND ≥50 chars. Gate: `gate:"human"`→`awaiting_approval`; `org_decision("no-go")`→run `halted`. `haltOn` field exists but no-go halts unconditionally today (fine — no-go halts).

**Protocol:** per task — fresh implementer, spec review, quality review; Important findings fixed + re-reviewed; TDD (fixture-based for tools). Separate commits. `tracked-followups.md` continues. Wave-close adversarial review before merge (the tools + the block path are the risk surface).

### W5.1 — `privacy_manifest_check` + `ats_check` compliance tools

**Files:** new `src/kilocode/tool/privacy-manifest-check.ts` + `.txt`, new `src/kilocode/tool/ats-check.ts` + `.txt`; wire both in `src/kilocode/tool/registry.ts` (`infos()`+`build()`); new permission keys; `test/kilocode/tool/privacy-manifest-check.test.ts` + `ats-check.test.ts`.
- `privacy_manifest_check` params `{ manifestPath: string, requiredReasonAPIs?: string[] }`: read the file at manifestPath (an XML `PrivacyInfo.xcprivacy` plist). Parse (self-contained minimal XML-plist walk for the specific keys — NO npm dep): extract `NSPrivacyAccessedAPITypes` (each entry's `NSPrivacyAccessedAPIType` + non-empty `NSPrivacyAccessedAPITypeReasons`) and `NSPrivacyTracking`. Return `{ ok, status: "ok"|"missing_manifest"|"invalid", violations: [{api?, key?, message}], rawPath }`. Violations: file missing → `missing_manifest`; a `requiredReasonAPIs` entry with no matching declared type+reason → a violation; malformed XML → `invalid` with a clear message. Fixture tests: a compliant manifest (ok:true, []); a manifest declaring an API but with EMPTY reasons → violation; missing file → `missing_manifest`; a required API not declared → violation. Ship the fixtures under `test/kilocode/tool/fixtures/`.
- `ats_check` params `{ plistPath: string }`: read an `Info.plist` (XML plist). Parse the `NSAppTransportSecurity` dict → flag `NSAllowsArbitraryLoads=true`, `NSAllowsArbitraryLoadsInWebContent=true`, and any `NSExceptionDomains` entry with `NSExceptionAllowsInsecureHTTPLoads=true`. Return `{ ok, violations: [{domain?, key, message}], rawPath }` (ok:true when no insecure setting). Fixtures: a compliant plist (no ATS or a strict one → ok); one with arbitrary loads → violation; one with an insecure exception domain → violation; missing key → ok (default-secure).
Both tools: no process spawn (pure file read+parse); `ctx.ask` on their own permission key; return the `{title, output: JSON, metadata}` shape the existing tools use. Report the minimal-plist-parse approach.

### W5.2 — `secret_scan` compliance tool (hardcoded-secret / guideline scan)

**Files:** new `src/kilocode/tool/secret-scan.ts` + `.txt`; wire in registry; new permission key; `test/kilocode/tool/secret-scan.test.ts` + fixtures.
- `secret_scan` params `{ paths: string[] }` (files or globs — resolve to files; cap total bytes read, e.g. 2MB, to avoid OOM — mirror the W2 bounded-read discipline). Scan each file's text for hardcoded-secret patterns: AWS keys (`AKIA[0-9A-Z]{16}`), generic `(api[_-]?key|secret|token|password)\s*[:=]\s*["'][^"']{8,}["']`, private-key headers (`-----BEGIN (RSA |EC )?PRIVATE KEY-----`), high-entropy base64-ish literals over a length threshold (conservative — avoid false positives on hashes/UUIDs; document the heuristic). Return `{ ok, findings: [{file, line, kind, snippet(redacted)}], filesScanned, bytesScanned }` (ok:true when findings empty). Redact the secret in the snippet (show first/last few chars). Fixtures: a clean Swift file (ok:true); a file with a hardcoded `apiKey = "sk-live-...."` and an AWS key → findings with file+line+kind; a binary/oversized file skipped gracefully. This tool detects the exit test's "hardcoded secret" defect. Report the false-positive guard.
