# Wave 4 — Workflow DAG Engine Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Branch: `feat/wave-4-dag` (off main; Waves 0-3 merged). `bun` at `~/.bun/bin`; never stage `bun.lock`; broad sweeps via `bun run script/test-runner.ts`; clean `"$(getconf DARWIN_USER_TEMP_DIR)"opencode-test-*` after runs. This wave generalizes the org **runner** — the most subtle, highest-risk code in the system (the v1 write-path Critical lived here). TDD is mandatory; every runner change lands with a failing test first.

**Goal:** Replace the flat linear pipeline with a dependency **DAG** so independent stages (frontend ∥ backend) run concurrently, stages can be conditionally skipped, and per-stage timeouts bound transient hangs — cutting wall-clock ~40% while preserving every existing budget/gate/retry/revise invariant.

**Architecture:** Keep the runner **pure and single-CEO-driven** (the W0-R2 invariant): the runner computes the *ready set* of stages and validates completed ones; the tool/CEO layer does the spawning. `advance` graduates from returning one action to returning a **batch** (multiple ready `instruct`s the CEO spawns as parallel `task` calls in one turn, plus at most one blocking gate/incomplete/halt). Stage readiness = all `requires` completed-or-skipped. `requires` **defaults to the immediately-preceding pipeline entry**, so existing linear org-templates behave byte-identically (opt-in parallelism only where a stage explicitly declares broader `requires`).

**Tech Stack:** Bun + zod v4 schema, the existing `OrgSchema`/`OrgState`/`OrgRunner`/`OrgArtifacts`/`OrgAudit`/`tools.ts` namespaces under `packages/opencode/src/kilocode/organization/`, org-template jsonc.

**Locked design decisions (grounded in the current runner, `runner.ts:250-436`):**
1. **Fan-out, not internal scheduler.** `advance` returns `{ instruct: [...], gate?, incomplete?, halted?, done? }`. The runner never spawns; the CEO spawns the returned instructs in parallel and calls `org_advance` again with all resulting `task_id`s. Preserves runner purity + testability.
2. **`requires` default = previous pipeline entry.** `resolveRequires(org)` fills an absent `requires` with `[prevStage]` (first stage → `[]`). Existing linear pipelines stay fully sequential; parallelism is opt-in.
3. **`maxConcurrency` (org-level, default 1).** The runner emits at most `maxConcurrency` instructs per batch. Default 1 = current sequential behavior even for DAGs; a template sets it >1 to actually parallelize. This makes the whole wave a no-op for existing orgs until they opt in — the safest possible migration.
4. **`when` is declarative, not a DSL (v1).** `when?: { mode?: string } | { stage: string; decision: "approve"|"no-go"|"revise" }`. Evaluated against a run-level `mode` (set at `org_start`) or a prior stage's recorded decision. A stage whose `when` is false is marked `"skipped"` and counts as satisfied for its dependents. Full expression language is v2 (dossier §D).
5. **`timeoutMs` is poll-checked.** The runner is driven by CEO polling, so a running stage whose `startedAt` is older than `timeoutMs` (and still un-validated on an `advance` call) routes through the existing `retryOrFail` with a `"timeout"` cause. No background timers (keeps the runner pure/synchronous-composable).
6. **Gates/incompletes still serialize.** With parallelism, `advance` may find several running stages; it validates all, but if any needs a human gate or is incomplete, it surfaces ONE such blocker per call (deterministic order: pipeline index). Independent ready stages still fan out alongside. Budget ceilings sum across all concurrent stages (unchanged math, `runTotal` already sums all stages).

**Protocol:** per task — fresh implementer, spec review, quality review; Important findings fixed + re-reviewed; TDD (failing test first, real output). Separate commits. The prior waves' `tracked-followups.md` ledger continues.

### W4.1 — Schema: DAG fields + `resolveRequires` + cycle validation

Extend `OrgSchema.Stage` with `requires: z.array(z.string().min(1)).optional()`, `timeoutMs: z.number().int().positive().optional()`, and `when: z.union([...]).optional()` (the two shapes from decision #4). Add `maxConcurrency: z.number().int().positive().optional()` to `Organization`. Add `export function resolveRequires(org): Record<string, string[]>` implementing decision #2 (absent → `[prevStage]`, first → `[]`; an explicit `[]` stays `[]` — an explicit root). Extend `validate(org)` with: every `requires` entry references a real pipeline stage; **cycle detection** over the resolved requires graph (DFS/Kahn — report the cycle path); a `when.stage` references a real, *earlier-or-unordered* stage; `maxConcurrency >= 1`. Keep all existing validation. Tests: resolveRequires defaulting (linear → each depends on prev), explicit requires preserved, explicit `[]` root; cycle rejected with path; dangling `requires`/`when.stage` rejected; back-compat (a pipeline with no new fields resolves to the linear chain). **No runner behavior change in this task** — schema + pure helpers only.

These selectors are the DAG's core logic — test them exhaustively at the pure level (no runner, no disk).

### W4.3 — Runner: batch `advance` (fan-out ready set, multi-running validation)

**Files:** `runner.ts` (`Advance` → `Batch` result; `advance`; `priorDeliverables` DAG-aware), `test/kilocode/organization/runner.test.ts`.

Introduce `export type Batch = { instruct: InstructItem[]; gate?: GateItem; incomplete?: IncompleteItem; halted?: { reason: string }; done?: true }` (reuse the existing per-item shapes). Rewrite `advance` to: (1) short-circuit halted/completed/failed as today; (2) validate EVERY `runningStages` entry that carries a `taskID` this call — the existing per-stage completion logic (cost recording, budget ceiling, escalation-once, revise-staleness, retryOrFail) runs **per validated stage, unchanged**; the first stage that must gate/incomplete/halt is surfaced as the batch's single `gate`/`incomplete`/`halted` (deterministic pipeline-index order); (3) compute `readyStages`, mark up to `maxConcurrency − (currently running count)` of them `running` (+`startedAt`,+`attempts`), and return them as `instruct[]`; (4) if nothing running/ready/gated and no pending remain, `done`. Make `priorDeliverables` DAG-aware: the transitive `requires` closure of the target stage (completed deliverables only), not pipeline-array-prefix. **Preserve every W0-W1 invariant** — reuse `retryOrFail`, `stageCost`, the `OrgState.update` atomic cost+budget block verbatim per stage. Tests: linear pipeline drives identically to the pre-wave runner (regression pin — same instruct sequence); a diamond with `maxConcurrency: 2` fans out frontend+backend in one batch, both validated on the next call, `integrate` instructed after; run-ceiling halt when concurrent stages' summed cost exceeds `budget.run`; a human gate on one branch surfaces while the independent branch still fans out; `maxConcurrency: 1` on a diamond stays sequential. This is the wave's riskiest task — the regression pin against today's linear behavior is the primary safety net.

### W4.4 — Conditional `when`: skip stages

**Files:** `runner.ts` (readiness → skip evaluation), `state.ts` (run-level `mode?`), `tools.ts` (`org_start` accepts `mode`), `test/kilocode/organization/runner.test.ts`.

Persist an optional run-level `mode: string` on `OrgState.Run` (set at `org_start`, default undefined). When `advance` is about to mark a `ready` stage `running`, first evaluate its `when` (decision #4): `{ mode }` → true iff `run.mode === when.mode`; `{ stage, decision }` → true iff that stage's recorded `decision === when.decision`. If `when` is present and false, mark the stage `"skipped"` (not `running`, no instruct, no cost) and re-run readiness (its dependents may now be ready). Tests: `mode: "mvp"` skips a `when:{mode:"full"}` marketing stage and its dependents still resolve; a `when:{stage:"evaluation",decision:"approve"}` stage runs on approve, skips on no-go path; a skipped stage never produces an instruct and never appears in cost. `org_status`/detail already surface `skipped` via the status field (Wave 3 view tolerates unknown → verify the console badge maps it; add mapping if missing — cross-check `packages/kilo-console/src/routes/orgs/org-runs-view.ts`).

### W4.5 — Per-stage timeout policy

**Files:** `runner.ts` (timeout check in the running-stage path), `test/kilocode/organization/runner.test.ts`.

In `advance`, for each running stage carrying a `taskID` whose deliverable is NOT yet valid, if `stage.timeoutMs` is set and `now − Date.parse(startedAt) > timeoutMs`, route to `retryOrFail` with a new `cause: "timeout"` (message: `stage "X" exceeded its Nms timeout without a valid deliverable`) instead of the plain `never-produced` incomplete — so a hung stage consumes a retry and eventually fails the run rather than looping forever. Inject `now` (a `() => number` in `Deps`, defaulting to `Date.now`) so the test is deterministic. Tests: a stage past its timeout with no deliverable → retryOrFail timeout path (retries then halts); a stage that completes before timeout is unaffected; no `timeoutMs` set → today's behavior (never times out). Keep `retryOrFail`'s existing cost-recording-on-retry.

Keep the mutex (`withRunLock`) around the whole mutating body. Tests: a batch of 2 instructs surfaces as 2 task calls; passing 2 task_ids validates both stages; a single instruct still works (back-compat); the CEO template mentions parallel spawning; mutex still wraps org_advance.
